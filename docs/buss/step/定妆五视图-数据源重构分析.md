## 🎯 重构目标

### 数据流重构

```
前端请求
    │
    ├── projectId (必填)
    ├── allInOneSlot (可选)
    └── referenceImages (可选，用于图生图参考)

    │       └── 构建 phase1Outfit
    │
    └── 4. 使用数据库数据构建提示词
```

## 📝 核心问题：数据是否完整？

### 问题 1：角色表的性别字段从哪里来？

**来源**：Step1 人像检测接口 `/library/characters/check-portrait`

```typescript

      gender: "male" | "female" | "other",  // ← LLM 识别的性别
      age: "25",
```

**问题**：
- 用户上传人像后，LLM 会自动识别性别
- 但如果 LLM 识别错误，或者用户未上传人像，性别字段可能为空或错误

## 🔍 数据完整性检查

### 场景 1：用户正常流程

```
Step1: 上传人像 → 人像检测 → 识别性别 (gender: "male")
     → 上传服饰 → 服饰分类 → 生成描述
     → 选择搭配方案 → 确认

Step2: 调用 generate-five-view
     → 从数据库读取角色信息 (gender: "male") ✅
     → 从数据库读取服饰信息 (description: "...") ✅
     → 构建提示词 ✅
```

### 场景 2：用户跳过某些步骤

```
Step1: 上传人像 → 未做人像检测 → gender 为空 ❌
     → 上传服饰 → 未做分类 → description 为空 ❌

Step2: 调用 generate-five-view
     → 从数据库读取角色信息 (gender: null) ❌
     → 从数据库读取服饰信息 (description: null) ❌
     → 无法构建有效提示词 ❌
```

## 💡 重构方案

### 方案 A：完全依赖数据库（严格模式）

**优点**：
- 数据来源单一，可追溯
- 强制用户完成 Step1 的所有必要步骤

**缺点**：
- 需要确保 Step1 数据完整性
- 如果数据缺失，接口会失败

**实现**：

```typescript
// Step 1: 查询项目关联的选中角色
const selectedCharacter = await ctx.projectCharacterService.getSelected(project.id);
if (!selectedCharacter) {
  throw new AppError(400, "NO_SELECTED_CHARACTER", "请先在 Step1 选择角色");
}

// Step 2: 查询角色详情
const character = await ctx.repos.libraryCharacters.findById(selectedCharacter.libraryCharacterId);
if (!character) {
  throw new AppError(404, "CHARACTER_NOT_FOUND", "角色不存在");
}

// Step 3: 构建核心特征（从角色表字段）
const coreFeatures = buildCoreFeaturesFromCharacter(character);

// Step 4: 查询项目服饰
const projectGarments = await ctx.repos.projectGarments.findByProjectId(project.id);
const phase1Outfit = buildPhase1OutfitFromGarments(projectGarments, ctx.repos.garmentAssets);
```

---

### 方案 B：数据库优先 + 前端兜底（兼容模式）

**优点**：
- 向后兼容现有流程
- 允许前端传递数据作为兜底

**缺点**：
- 数据来源不单一
- 需要处理冲突情况

**实现**：

```typescript
// 优先从数据库读取
let coreFeatures = null;
let phase1Outfit = null;

// 尝试从角色表读取
const selectedCharacter = await ctx.projectCharacterService.getSelected(project.id);
if (selectedCharacter) {
  const character = await ctx.repos.libraryCharacters.findById(selectedCharacter.libraryCharacterId);
  if (character?.gender) {
    coreFeatures = buildCoreFeaturesFromCharacter(character);
  }
}

// 尝试从服饰表读取
const projectGarments = await ctx.repos.projectGarments.findByProjectId(project.id);
if (projectGarments.length > 0) {
  phase1Outfit = await buildPhase1OutfitFromGarments(projectGarments, ctx.repos.garmentAssets);
}

// 如果数据库数据不完整，使用前端传参作为兜底
if (!coreFeatures && body.coreFeatures) {
  coreFeatures = body.coreFeatures;
}
if (!phase1Outfit && body.phase1Outfit) {
  phase1Outfit = body.phase1Outfit;
}

// 最终校验
if (!coreFeatures) {
  throw new AppError(400, "MISSING_CORE_FEATURES", "缺少角色核心特征数据");
}
```

### 方案 C：前端传递角色 ID + 后端查询（推荐）

**优点**：
- 前端只需传递 ID，不传递大段文本
- 后端统一管理数据转换逻辑
- 数据来源清晰

**缺点**：
- 需要修改前端接口调用

**实现**：


## 📋 核心转换函数设计

### 1. 从角色表构建 coreFeatures

```typescript
/**
 * 从角色表字段构建英文核心特征描述
 */
function buildCoreFeaturesFromCharacter(character: LibraryCharacter): string {
  // 复用现有的映射函数
  const preset: Step1RolePreset = {
    overallImpression: character.overallImpression || "clean and natural",
    ethnicityOrRegion: character.ethnicity || "Asian",
    gender: character.gender || "unknown",
    age: parseInt(character.age || "20"),
    styleWords: character.style?.split(/[,，、]/).map(s => s.trim()).filter(Boolean) || ["natural"],
  };
  
  return mapStep1RolePresetToEnglishCoreFeatures(preset).coreFeatures;
}
```

### 2. 从服饰表构建 phase1Outfit

```typescript
/**
 * 从项目服饰构建穿搭描述
 */
async function buildPhase1OutfitFromGarments(
  projectGarments: ProjectGarment[],
  garmentRepo: IGarmentAssetRepository,
): Promise<string> {
  const descriptions: string[] = [];
  
  for (const pg of projectGarments) {
    if (!pg.garmentAssetId) continue;
    
    const garment = await garmentRepo.findById(pg.garmentAssetId);
    if (!garment) continue;
    
    // 组合服饰描述
    const parts = [
      garment.name,
      garment.description,
      garment.mainColor ? `${garment.mainColor} color` : null,
      garment.material ? `${garment.material} material` : null,
      garment.style ? `${garment.style} style` : null,
    ].filter(Boolean);
    
    descriptions.push(`${garment.category}: ${parts.join(", ")}`);
  }
  
  return descriptions.join(". ");
}
```

## 🚨 风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 角色表性别字段为空 | 高 | Step1 强制要求人像检测，或提供手动选择性别功能 |
| 服饰描述为空 | 中 | 使用服饰名称 + 分类信息兜底 |
| 数据不一致 | 中 | 增加数据校验逻辑，记录数据来源 |
| 性能影响 | 低 | 增加数据库查询，但可缓存 |

## 📌 建议

### 短期（推荐方案 B）

1. **保持接口兼容**：支持前端传递 `coreFeatures` 和 `phase1Outfit`
2. **优先读取数据库**：如果项目有关联角色，从角色表构建 `coreFeatures`
3. **记录日志**：记录数据来源（数据库/前端），便于排查问题

### 中期（推荐方案 C）

1. **前端只传 ID**：`characterId` 和 `outfitPlanId`
2. **后端统一构建**：所有文本描述由后端生成
3. **数据完整性校验**：Step1 必须完成必要的数据采集

### 长期

1. **Step1 数据规范**：定义必填字段和校验规则
2. **数据版本管理**：记录数据变更历史
3. **数据质量监控**：统计字段缺失率和错误率
