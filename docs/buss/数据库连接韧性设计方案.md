# 数据库连接韧性设计方案

## 问题现象

数据库连接失败（`ECONNREFUSED 101.37.80.207:5432`）后导致进程崩溃退出。

## 错误链路

```
数据库连接失败 (ECONNREFUSED)
    ↓
[1] job-executor 持续报错 (每秒多次)
    └─ safeProcessOnce() 捕获错误只记录日志，不中断定时器
    ↓
[2] step3 preview job 执行失败
    └─ 进入 catch 块处理错误
    ↓
[3] persistStep3PreviewJobStatusToWorkflowState() 再次尝试数据库操作
    └─ 数据库仍然断开，抛出 ECONNREFUSED
    ↓
[4] Unhandled rejection
    └─ catch 块内的异步操作失败，外层无法捕获
    ↓
[5] server.ts:36 触发 process.exit(1)
```

## 根因分析

### 问题一：全局错误处理器缺少 `ECONNREFUSED` 处理

`server.ts:14` 只处理了 `Connection terminated` 和 `ECONNRESET`，但 `ECONNREFUSED` 没有处理。

### 问题二：错误处理代码本身会失败

`step3-candidate-helpers.ts:2149-2159` 中，在 `catch` 块内调用 `persistStep3PreviewJobStatusToWorkflowState()`，如果数据库仍然断开，这个操作也会失败。

### 问题三：连接池没有重连机制

`startup-pg-pool.ts:32` 创建的 `Pool` 没有配置错误事件监听器。

## 待决策问题

**当数据库连接恢复后，期望的行为是什么？**

| 选项 | 说明 |
|------|------|
| **A. 自动恢复** | 进程继续运行，连接池自动重连，后续操作正常工作 |
| **B. 快速失败** | 进程立即退出，由外部进程管理器（如 PM2/Docker）重启 |
| **C. 降级运行** | 进程继续运行，但标记服务为"不可用"状态，拒绝新请求 |

## 候选方案

### 方案 A：快速修复（最小改动）

在 `server.ts` 全局错误处理器中添加 `ECONNREFUSED` 处理，防止进程退出。

**优点**：改动小，风险低
**缺点**：只是治标，不解决根本问题

### 方案 B：完整修复（推荐）

A + 在错误处理代码中添加二次 try-catch + 连接池配置优化

**优点**：从根本上解决问题，增强系统韧性
**缺点**：改动范围较大

### 方案 C：架构优化

将后台任务的错误持久化改为队列模式（内存队列 + 重试机制），解耦数据库依赖

**优点**：彻底解决后台任务失败影响主进程的问题
**缺点**：需要引入新的基础设施
