# 异步任务队列可靠性改进设计

## 问题背景

### 触发场景

项目 `8597bc50-28b4-4384-b606-c1991eba4009` 在 Step3 分镜图生成过程中，只生成了 2 张图片，后续 3 张未自动生成。经排查，原因是批量生成队列被中断（可能原因：服务器重启、页面刷新、网络问题等），前端无法自动恢复。

### 根因分析

**数据流追踪：**

```
用户点击"生成分镜预览"
    ↓
确认锁定脚本 → SCRIPT_CONFIRMED
    ↓
生成 shot_prompts → 5 个镜头 ✅
    ↓
启动批量队列（threadCount=2）
    ↓
    ├─ Worker 1: frame_index=1 ✅ succeeded
    └─ Worker 2: frame_index=2 ✅ succeeded
    ↓
[中断点] 服务重启/页面刷新/网络问题
    ↓
    ├─ frame_index=3 未启动 ❌
    ├─ frame_index=4 未启动 ❌
    └─ frame_index=5 未启动 ❌
```

**关键发现：**

| 数据层 | 预期 | 实际 | 状态 |
|--------|------|------|------|
| `nrm_project_workflow_states.project_data.script` | 5 个镜头 | 5 个镜头 | ✅ 正常 |
| `nrm_shot_prompts.shots` | 5 个提示词 | 5 个提示词 | ✅ 正常 |
| `nrm_step3_frame_jobs` | 5 条任务 | 2 条任务 | ❌ 异常 |
| `nrm_step3_frame_images` | 2 张图片 | 2 张图片 | ✅ 与 jobs 一致 |

## 方案对比

### 方案 A：页面加载时自动恢复（最小改动）

**原理：** 在 Step3 页面加载时，检测是否有未完成的分镜图任务，自动启动批量生成。

**改动点：**
- 前端 `ScriptEditor.tsx` 增加一个 `useEffect`
- 检测 `pendingCount > 0` 且脚本已确认，自动调用批量生成

**优点：**
- 改动最小（约 20 行代码）
- 利用现有逻辑，无需新增后端 API

**缺点：**
- 用户不刷新页面则不会恢复
- 服务器重启后，前端无感知

**覆盖场景：**
| 场景 | 是否覆盖 |
|------|---------|
| 用户刷新页面后恢复 | ✅ |
| 用户不做操作，自动继续 | ❌ |
| 服务器重启后恢复 | ❌ |
| 用户关闭浏览器后继续 | ❌ |

---

### 方案 B：后端队列服务（推荐）

**原理：** 引入 Redis + BullMQ，将任务持久化到队列，独立 Worker 进程执行。

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   前端      │ ──▶ │   后端 API  │ ──▶ │   Redis     │
│  触发生成   │     │  创建任务   │     │  任务队列   │
└─────────────┘     └─────────────┘     └─────────────┘
                                               │
                                               ▼
                                        ┌─────────────┐
                                        │  Worker     │
                                        │  执行生成   │
                                        └─────────────┘
```

**优点：**
- 任务持久化，服务重启后自动恢复
- 可暂停/恢复/重试
- 支持优先级、延迟执行
- 可跨服务实例协调

**缺点：**
- 需要引入 Redis
- 改动较大
- 需要部署独立的 Worker 进程

**覆盖场景：**
| 场景 | 是否覆盖 |
|------|---------|
| 用户刷新页面后恢复 | ✅ |
| 用户不做操作，自动继续 | ✅ |
| 服务器重启后恢复 | ✅ |
| 用户关闭浏览器后继续 | ✅ |

---

### 方案 C：浏览器后台 Worker

**原理：** 使用 Web Worker 或 Service Worker 在后台执行任务。

**优点：**
- 用户切换页面不影响
- 不需要后端改动

**缺点：**
- 浏览器兼容性问题
- 用户关闭浏览器仍会中断
- 前端复杂度增加
- 无法解决服务器重启问题

**覆盖场景：**
| 场景 | 是否覆盖 |
|------|---------|
| 用户刷新页面后恢复 | ✅ |
| 用户不做操作，自动继续 | ✅ |
| 服务器重启后恢复 | ❌ |
| 用户关闭浏览器后继续 | ❌ |

## 改造策略建议

### 渐进式改造路线

**推荐策略：设计统一架构，分批迁移**

| 阶段 | 内容 | 优先级 |
|------|------|--------|
| **阶段 1** | 设计通用队列服务基础设施 | 高 |
| **阶段 2** | 迁移 Step3 分镜图生成（最常遇到中断） | 高 |
| **阶段 3** | 迁移 Step5 视频生成 | 中 |
| **阶段 4** | 迁移 Step3 脚本生成、脚本反推 | 低 |
| **阶段 5** | 迁移裂变视频、五视图生成 | 低 |

### 队列服务核心接口

```typescript
// 统一的任务类型
interface JobPayload {
  jobType: string;        // step3_frame | step5_video | step3_script | ...
  projectId: string;
  userId: string;
  data: Record<string, unknown>;
}

// 统一的任务状态
interface JobStatus {
  jobId: string;
  status: 'pending' | 'running' | 'succeeded' | 'failed';
  progress?: number;      // 0-100
  result?: unknown;
  error?: { code: string; message: string };
}

// 队列服务接口
interface IQueueService {
  // 创建任务
  createJob(payload: JobPayload): Promise<string>;
  
  // 查询任务状态
  getJobStatus(jobId: string): Promise<JobStatus>;
  
  // 取消任务
  cancelJob(jobId: string): Promise<void>;
  
  // 重试任务
  retryJob(jobId: string): Promise<void>;
}
```

## 待决策问题

### 1. Redis 部署方式

| 选项 | 说明 |
|------|------|
| A. 单机 Redis | 简单，适合开发和小规模部署 |
| B. Redis Sentinel | 高可用，适合生产环境 |
| C. 云服务（阿里云 Redis） | 无需运维，按量付费 |

### 2. Worker 部署方式

| 选项 | 说明 |
|------|------|
| A. 与 API 同进程 | 简单，但 API 重启会影响任务 |
| B. 独立进程 | 推荐，任务执行不受 API 影响 |
| C. 独立服务 | 可水平扩展，适合高并发场景 |

### 3. 迁移优先级

| 选项 | 说明 |
|------|------|
| A. 只改造 Step3 分镜图 | 最小范围 |
| B. 同时改造 Step3 分镜图 + Step5 视频 | 覆盖主要场景 |
| C. 设计统一架构，逐步迁移全部 | 推荐 |
