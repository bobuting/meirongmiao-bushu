# 64. AppShell 扩展注册口接入说明 - 2026-03-25

## 1. 本次调整结论

主仓没有采用“继续在 `src/app.ts` 末尾堆 `otherHandlers` 逻辑”的做法，而是做成了正式 seam：

1. `src/routes/app-shell-thin-entry.ts`
   - 新增 `extraRegistrars?: readonly AppShellExtraRouteRegistrar[]`
   - 保留原有 3 个冻结域不变
   - 在冻结域注册完成后，再顺序注册 `extraRegistrars`
2. `src/routes/app-shell-handlers.ts`
   - 新增 `createAppShellHandlers(ctx, requireUser, requireAdmin)`
   - 当前默认返回空的 `extraRegistrars`
   - 后续 donor agent 只需要改这里，不要再回头扩 `src/app.ts`
3. `src/app.ts`
   - 仅新增一行 import
   - 仅新增一行 `const otherHandlers = createAppShellHandlers(...)`
   - 仅在 `registerAppShellThinEntry(...)` 调用处 spread 一次 `...otherHandlers`

一句话：**主仓已经预留好了 app-shell 扩展口，后续新增模块路由应接到 `app-shell-handlers.ts` 的 `extraRegistrars`，而不是继续膨胀 `src/app.ts`。**

---

## 2. 现在 donor agent 应该怎么接

如果外仓有新的 app-shell 路由要并进主仓，按下面做：

1. 在 `src/routes/app-shell-handlers.ts` 中补 `extraRegistrars`
2. 每个新增域对应一个 `id`
3. 每个新增域在 `register(app, ctx)` 中只负责注册自己的 route seam
4. 不要修改 `src/app.ts` 除非只是 import 或一行接线

推荐形态：

```ts
export function createAppShellHandlers(
  ctx: AppContext,
  requireUser: ...,
  requireAdmin: ...,
): AppShellHandlersBundle {
  return {
    extraRegistrars: [
      {
        id: "your_new_domain_routes",
        register: (app, innerCtx) => {
          registerYourNewDomainRoutes(app, {
            ctx: innerCtx,
            requireUser,
            requireAdmin,
          });
        },
      },
    ],
  };
}
```

---

## 3. 必须遵守的边界

1. 不要改 `src/routes/index.ts` 里冻结的 3 个 registrar id
2. 不要把新域硬塞回 `project_flow_routes`，除非它本来就属于 Step1-Step5 主流程
3. 不要在 `src/app.ts` 继续追加 route 逻辑
4. `extraRegistrars` 的 `id` 不能与以下保留值冲突：
   - `project_flow_routes`
   - `reverse_square_routes`
   - `admin_library_provider_routes`
5. 同一个 `extraRegistrars` 列表内，`id` 不能重复

---

## 4. 这样做的原因

这样处理有 4 个直接收益：

1. 解决“外仓新增 app-shell 路由无处挂载”的真实问题
2. 不破坏主仓对 `src/app.ts` 的 hotspot 治理
3. 不污染现有 `registerAppShellThinEntry` 的 3 个冻结主域
4. 给后续 donor agent 留下一个稳定、明确、单一的接入口

---

## 5. 对 donor agent 的直接说明

后续如果你要补自己的 app-shell 路由：

1. **改 `src/routes/app-shell-handlers.ts`**
2. **不要再要求主仓在 `src/app.ts` 末尾继续手工拼接新的 handler 对象**
3. **如果你的路由属于 Step1-Step5 主流程，那就优先进已有 registrar，而不是走 extraRegistrars**

当前主仓已经把“可扩展接入口”准备好了，后续请沿着这个 seam 接，不要重开热点口。
