## Context

换装项目（outfit_change）是一个 4 步工作流：选择视频 → 选择服装 → 选择角色 → 一键换装。

**问题1：步骤导航**
当前所有步骤的底部工具栏都只有"返回我的项目"按钮，用户无法在步骤间回退。

**问题2：内容分页**
Step 2 服装列表和 Step 3 角色列表一次性全部展示，当用户有大量服装/角色时，页面过长，难以浏览。

**技术背景：**
- 前端使用 React 18 + TypeScript + Tailwind CSS
- 页面位于 `apps/web/pages/outfit-change/`
- 导航使用 React Router (`useNavigate`)
- 数据通过 TanStack Query 获取

## Goals / Non-Goals

**Goals:**
- Step 2-4 底部工具栏左侧按钮改为"上一步"
- Step 3 新增"跳过"按钮
- Step 2/3 列表添加分页功能（每页 12 个）
- 保持 Step 1 的"返回我的项目"按钮不变

**Non-Goals:**
- 不修改后端 API（分页在前端实现，数据仍一次性获取）
- 不修改视频项目的导航逻辑
- 不添加新的数据持久化逻辑

## Decisions

### 1. 步骤导航设计（已实现）

```
Step 1: [返回我的项目] [状态文本] [下一步]
Step 2: [上一步] [状态文本] [下一步]
Step 3: [上一步] [跳过] [状态文本] [下一步]
Step 4: [上一步] [状态文本] [开始换装]
```

### 2. 分页实现设计

**决策：前端分页（不修改后端 API）**

理由：
- 服装/角色数据量通常不大（几十到几百个）
- 后端 API 已返回完整列表，前端分页更简单
- 避免修改后端 API 的成本

**分页参数：**
- 每页显示数量：12 个（4 列 × 3 行）
- 状态管理：使用 `useState` 存储 `currentPage`
- 数据切片：`items.slice((page - 1) * 12, page * 12)`

**翻页控件设计：**
```
[< 上一页]  页码/总页数  [下一页 >]
```

- 位置：列表底部，居中显示
- 样式：Material Icons + 文字按钮
- 响应式：移动端仅显示图标

### 3. 分页状态保留

**问题：** 用户翻页后选择商品，是否保持页码？

**决策：** 保持当前页码，不重置到第一页

理由：用户可能在某一页浏览多个商品，保持页码减少操作成本。

### 4. 分页组件位置

**决策：** 分页控件放在内容区域内，不放在底部浮动工具栏

理由：
- 底部工具栏用于步骤导航，不应混入内容分页
- 分页控件紧贴列表下方更符合用户直觉

## Risks / Trade-offs

### 风险 1：前端分页在数据量大时性能问题

**风险描述：** 如果用户有上千个服装/角色，一次性获取数据可能较慢。

**缓解措施：**
- 当前场景下用户数据量通常不大
- 后续如需优化可改为后端分页 API

### 风险 2：跳过 Step 3 后可能影响换装效果

**风险描述：** 如果用户跳过角色选择，换装效果可能使用原视频人物。

**缓解措施：** Step 4 状态文本提示"未选择角色，将使用原视频人物"

## Open Questions

1. **每页数量是否合适？** 12 个（4 列 × 3 行）可根据实际效果调整
2. **是否需要"跳转到第一页/最后一页"按钮？** 建议先实现简单翻页，后续按需添加
